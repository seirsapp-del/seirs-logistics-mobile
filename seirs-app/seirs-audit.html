Business / Partner App
⚠
Batch 4 Does Not Exist

The spec defines Batch 4 as a full standalone app with two user types (Partner Store owners, Business Senders) and substantial feature sets. The current architecture contains only a "Business" settings screen in the customer app that redirects to a sales page. Zero of §4.2 or §4.3 features are built: no bulk delivery, no CSV upload, no recurring schedules, no business wallet, no partner store dashboard, no specialist delivery booking, no team member roles.

Stack Assessment
✓
Tech Stack is Correct

NestJS + PostgreSQL + Redis + Cloudflare R2 + Firebase FCM is a well-proven combination. The monorepo with @seirs/shared is the right call for keeping theme tokens, API types, and models consistent across apps. The fundamental choices are sound; the gaps are in implementation depth, not architecture direction.

Single Points of Failure
ComponentRiskRecommended Fix
Railway single instanceEntire backend on one container. At ~1,000 concurrent WebSocket connections + API traffic, it will destabilize.Horizontal scaling via Railway services + Redis WebSocket adapter for Socket.IO
WebSocket gateway (single process)If the NestJS process restarts, all active deliveries lose real-time tracking. Redis pub/sub adapter not confirmed.Implement @nestjs/platform-socket.io with Redis adapter; confirm adapter is wired
No job queueEmails, push notifications, and payment webhook retries run synchronously in request cycles. A Flutterwave webhook failure drops the payment event permanently.Add BullMQ; move email, push, and webhook retry to async queues with exponential back-off
No read replicaAnalytics queries (revenue aggregations, heatmaps) compete with transactional writes on a single PostgreSQL instance. Degraded write latency at scale.Add Railway PostgreSQL replica; route all analytics queries to replica
No spatial indexingNearest-driver matching and partner store proximity are full table scans without PostGIS.Enable PostGIS extension; add geometry column to drivers table; use ST_DWithin queries

NOT PRODUCTION READY

Platform is approximately 38% complete relative to V7 specification

The foundation is technically sound. The stack choices are correct, the monorepo structure is right, and the core delivery flow works at small scale. But the platform has critical unresolved issues that make it unsuitable for public access at any user volume.

  • Maps — the platform's primary visual and functional feature — are broken across all apps
  • Wallet withdrawal is broken — drivers cannot be paid
  • There is no RBAC — any admin breach exposes everything to everyone
  • There is no 2FA on the admin dashboard handling live financial data
  • There is no audit trail for any admin action
  • There is no ticketing system — support team has zero tooling
  • Batch 4 (Business / Partner App) does not exist
  • Plaintext credentials are committed to the repository

Path to Industry Standard

To reach the level of DHL, Uber, or Wolt on the Nigerian market, the following capabilities are non-negotiable beyond spec compliance:

Must-Have for Competitiveness

  • Live map on home screen — strongest first-impression signal in logistics
  • Pricing engine as dynamic as any ride-hailing platform: zone-aware, fuel-indexed, surge-capable
  • Full i18n in Yoruba, Igbo, Hausa — genuine competitive moat; no competitor has executed this well in Nigeria
  • Booking in 3 steps maximum — each additional step costs conversion
  • Public web tracking URL — shareable with recipients who don't have the app
  • Driver earnings motivation features — goal tracker, weekly summaries, demand heatmap keep drivers active

Structural Investments for Scale

  • PostGIS on PostgreSQL — geo-queries are the core of driver matching
  • BullMQ job queue — all async operations (email, push, payments) must be decoupled
  • Redis WebSocket adapter — real-time tracking must survive multi-instance deployment
  • Sentry error tracking — invisible bugs in production are existential at scale
  • Read replica for analytics — protect transactional write performance
  • Backup + point-in-time recovery strategy before first external user
